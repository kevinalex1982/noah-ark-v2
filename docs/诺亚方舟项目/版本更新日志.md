## v1.0.34 (2026-04-19)

### 🐛 致命问题修复

- **第三步未操作就显示认证成功然后卡住**（根因修复）：palm/iris 轮询函数依赖 `countdown`，倒计时每秒变化导致 `useCallback` 每秒返回新引用，`useEffect` 随之每秒执行 cleanup → 重启轮询。等待越久堆积的 fetch 请求越多（40秒 = 40个请求），所有请求同时完成时主线程被阻塞，`router.push` 被严重延迟。修复：用 `countdownRef` 替代 `countdown` 依赖，使 polling 函数引用稳定，`useEffect` 不再因倒计时变化而反复重启
- **每步倒计时不重置**：掌纹认证耗时会影响下一步虹膜的倒计时时间。修复：① IdleTimer 新增 `resetKey` prop，步骤切换时自动重置 60 秒不操作返回倒计时；② 每步进入新步骤时重置 `countdown` 为 `initialAuthTimeout`（120 秒初始值）

**修改的文件**：
| 文件 | 修改内容 |
|------|----------|
| `app/kiosk/combined/page.tsx` | 新增 `countdownRef`、`initialAuthTimeout`；polling 函数改用 ref 读取倒计时；步骤切换时重置倒计时为初始值；IdleTimer 传 `resetKey` |
| `components/kiosk/IdleTimer.tsx` | 新增可选 `resetKey` prop，值变化时自动重置 60 秒倒计时 |

## v1.0.25 (2026-04-19)

### 🐛 致命问题修复

- **通行记录去重逻辑错误（时区问题）**：同一个人员不管间隔多久认证，都只会产生一条通行记录。根因是 `created_at` 存储的是北京时间（UTC+8），但去重查询 `datetime('now', '-8 hours')` 计算的是 UTC-8 的时间，时区差了 16 小时，导致所有今天的记录都被当作"重复"拦截。修复：查询改为 `datetime('now', '+8 hours')`，与存储的北京时间一致

### 🔧 其他修复

- **第三步认证成功后卡住不动**：`goToSuccess` 改为**先跳转再上传**，通行记录上传不再阻塞导航
- **第三步未操作就显示认证成功**：iris/palm polling 启动后延迟 3 秒再查询，给设备清理状态的时间
- **`goToSuccess` 被多次调用**：加 `goToSuccessCalledRef` 防护
- **认证超时时间太短**：默认 `authTimeout` 从 60 秒改为 120 秒

**修改的文件**：
| 文件 | 修改内容 |
|------|----------|
| `lib/db-pass-logs.ts` | `getRecentPassLogByPerson` 查询时区从 `-8 hours` 改为 `+8 hours` |
| `app/kiosk/combined/page.tsx` | `goToSuccess` 先跳转再上传；加重复调用防护；iris polling 延迟 3 秒 |
| `lib/settings.ts` | `authTimeout` 默认值 60→120 |

### 🐛 问题修复

- **通行记录 auth_type 重复**：组合认证成功后，上传的通行记录中 auth_type 出现大量重复（如"密码、虹膜、虹膜、虹膜…掌纹、掌纹…"）。根因是 `handleBiometricComplete` 被设备多次触发，`completedStepsRef` 累积了重复值。修复：① `handleBiometricComplete` 开头加防护，已完成的步骤直接跳过；② `goToSuccess` 用 `Set` 去重 authTypes 数组

## v1.0.23 (2026-04-18)

### 🐛 致命问题修复

#### 1. 组合认证通行记录从未上传到 IAMS

**问题描述**：组合认证成功后，通行记录从未写入数据库，也从未上报给 IAMS 服务器。

**根因**：
- 组合认证页面从 `/api/auth/types` 接口获取 `credentialId`
- 该接口**从未返回过 `credentialId` 字段**
- 页面用 `|| 0` 兜底，`credentialId` 始终为 0
- 上传 API 中 `!credentialId` 验证：`!0 === true` → 直接返回 400 拒绝

**修复内容**：
| 文件 | 修改 |
|------|------|
| `lib/database.ts` | `findByUserCode` SQL 查询增加 `credential_id` 字段 |
| `app/api/auth/types/route.ts` | API 响应新增 `credentialId` 字段 |
| `app/api/pass-log/upload/route.ts` | 验证改为 `credentialId === undefined \|\| === null`，允许值为 0 |
| `app/kiosk/combined/page.tsx` | 无条件设置 `userInfo`（不依赖 `personName` 是否有值） |

#### 2. 组合认证步骤判断不准确（闭包问题）

**问题描述**：3 凭证组合认证时，有时第一步验证完就通过了，有时第二步完就通过了，有时三步全完成才通过——行为不可预测。

**根因**：
- `handleBiometricComplete` 的 `useCallback` 依赖 `[steps, goToSuccess]`
- 该函数被 `startIrisPolling` 和 `startPalmPolling` 内联调用
- polling 函数的依赖数组是 `[identityId, countdown, stopPolling]`，**不包含 `handleBiometricComplete`**
- polling 函数中引用的 `handleBiometricComplete` 是创建时的旧闭包版本，其中读取的 `steps` 可能是空数组 `[]`
- `steps.indexOf(step)` 返回 `-1`，导致错误判断下一步或直接 `goToSuccess()`

**修复内容**：
| 文件 | 修改 |
|------|------|
| `app/kiosk/combined/page.tsx` | 新增 `stepsRef` 追踪最新 `steps`；新增 `goToSuccessRef` 追踪最新 `goToSuccess`；`handleBiometricComplete` 改为空依赖 `[]`，内部通过 ref 读取最新值 |

### 🔧 其他修复

- **组合认证日志路径调整**：从 `data/组合认证日志.log` 改为 `logs/combined-auth.log`，与 `startup.log`、`nextjs.log` 放在同一目录下（`%APPDATA%\noah-ark-electron\logs\`）
- **组合认证密码/胁迫码逻辑**：`hasPasswordType` 改为只判断 `includes(5)`，不再错误地将只有胁迫码(9)的情况当作需要密码
- **组合认证虹膜识别**：改为通过 `/api/device/iris/verify` 服务端比对，与单独认证虹膜保持一致
- **组合认证掌纹识别**：请求代码统一为字符串 `'103'`/`'102'`，`code` 比较改为 `String()` 转换，与单独认证掌纹完全一致
- **生物识别重试**：新增 `handleBiometricRetry` 函数，重试时重置倒计时
- **返回按钮**：组合认证返回改为跳转 `/kiosk` 首页

### 📝 新增功能

- **组合认证日志**：新增 `/api/combined-auth-log` API，记录组合认证每一步的状态变化
  - 记录：初始化步骤、密码提交、密码通过/失败、胁迫码触发、生物识别完成、跳转成功页面
  - 日志位置：`%APPDATA%\noah-ark-electron\logs\combined-auth.log`（与 `startup.log`、`nextjs.log` 同目录）

### 📄 文档更新

- 功能实现文档：新增 18.8 节（handleBiometricComplete 闭包修复）、18.9 节（组合认证日志）
