### 18.7 凭证 auth_model 保存修复（2026-04-18）

**问题描述**：
IAMS 下发的组合认证凭证（authModel=820）存入数据库后，认证类型显示为"单凭证识别"而非"多凭证组合识别"。

**根因**：
`handlePassportAdd` 中虹膜(type=7)、掌纹(type=8)、其他类型三个 `upsertCredential` 调用都**没有传入 `auth_model`** 字段，导致全部使用数据库默认值 1（单凭证）。

**修复内容**：
- 在 payload 类型中添加 `authModel?: number` 字段
- 虹膜新增 upsert 添加 `auth_model: payload.authModel`
- 掌纹新增 upsert 添加 `auth_model: payload.authModel`
- 其他类型 upsert 添加 `auth_model: payload.authModel`
- 凭证已存在时的 update 也添加 `auth_model: payload.authModel`

**auth_model 含义**：
| 值 | 含义 | 前端显示 |
|----|------|----------|
| 1 | 单凭证识别 | 单凭证 |
| 820 | 多凭证组合识别 | 组合 |

**修改的文件**：
| 文件 | 修改内容 |
|------|----------|
| `lib/device-sync.ts` | handlePassportAdd 4处 upsert/update 全部添加 auth_model |

> 最后更新：2026-04-18（组合认证掌纹与单独认证完全对齐、重试重置倒计时、auth_model 保存修复、handleBiometricComplete 闭包修复、组合认证日志）

### 18.8 组合认证 handleBiometricComplete 闭包修复（2026-04-18）

**问题描述**：
3 凭证组合认证时，有时第一步验证完就通过了，有时第二步完就通过了，有时三步全完成才通过——行为不可预测。

**根因**：
`handleBiometricComplete` 的 `useCallback` 依赖 `[steps, goToSuccess]`。但该函数被 `startIrisPolling` 和 `startPalmPolling` 内联调用，而这两个 polling 函数的依赖数组是 `[identityId, countdown, stopPolling]`——**不包含 `handleBiometricComplete`**。这意味着 polling 函数中引用的 `handleBiometricComplete` 是创建时的旧闭包版本，其中读取的 `steps` 可能是空数组 `[]` 或旧值。

当 `steps` 从 `[]` 变为 `['password', 'iris', 'palm']` 后：
- `handleBiometricComplete` 会重新创建（新闭包，`steps` 正确）
- 但 `startIrisPolling` 不会重新创建（`identityId`/`countdown`/`stopPolling` 都没变）
- 所以 `startIrisPolling` 内调用的 `handleBiometricComplete` 仍是旧版本，`steps.indexOf(step)` 返回 `-1`
- `-1 < -1 + 1` 即 `-1 < 0` 为 `true`，会尝试 `steps[0]` 作为下一步，或直接走 `goToSuccess()`

**修复方案**：
1. 新增 `stepsRef` 追踪最新的 `steps` 数组
2. 新增 `goToSuccessRef` 追踪最新的 `goToSuccess` 回调
3. `handleBiometricComplete` 改为空依赖 `[]`，内部通过 ref 读取最新值
4. 添加组合认证日志 API，记录每一步的状态变化

**修改的文件**：
| 文件 | 修改内容 |
|------|----------|
| `app/kiosk/combined/page.tsx` | 添加 stepsRef、goToSuccessRef，重写 handleBiometricComplete 使用 ref，添加日志调用 |
| `app/api/combined-auth-log/route.ts` | 新增日志 API，写入 data/组合认证日志.log |

### 18.9 组合认证日志（2026-04-18）

在组合认证流程的每个关键节点写日志到 `data/组合认证日志.log`，包括：
- 初始化步骤（authTypeList、总步骤数）
- 密码提交（验证开始）
- 密码通过/失败（当前步骤、已完成步骤、下一步动作）
- 生物识别完成（步骤名、当前位置、是否最后一步、下一步动作）
- 跳转成功页面（最终完成的步骤列表）

日志格式：
```
[2026-04-18 22:30:15] action: biometric_complete, step: iris, completedSteps: ["password","iris"], currentIndex: 1, totalSteps: 3, nextAction: goToNext
```

## ⚠️ 核心规则（必须遵守）

### 一、虹膜设备操作时序规范

#### 1.1 虹膜添加流程

```
锁定设备 → 等待8秒 → 上传人员(memberSave) → 等待500ms → 解锁设备
```

**时间要求：**
- 锁定后等待：**8秒**
- 上传后等待：**500ms**

**代码位置：** `lib/device-sync.ts` → `syncToIrisDevice()`

#### 1.2 虹膜更新流程

```
只更新数据库属性，不操作设备！
```

**⚠️ 重要变更：** `passport-update` 操作不再操作设备，只更新数据库中的属性字段

**代码位置：** `lib/device-sync.ts` → `handlePassportUpdate()`

#### 1.3 虹膜删除流程

```
直接调用 memberDelete，不需要锁定！
```

**注意：** 删除操作不需要锁定设备

**代码位置：** `lib/device-sync.ts` → `deleteFromIrisDevice()`

---

### 二、数据库操作时机（重要！）

#### 2.1 添加操作

```
先同步设备成功 → 再保存数据库
```

**错误做法：** 先保存数据库，后同步设备（设备失败时数据库有脏数据）

**正确做法：**
```typescript
// 1. 先同步设备
const result = await syncToIrisDevice(...);

// 2. 设备成功才保存数据库
if (result.success) {
  await upsertCredential(...);
}
```

#### 2.2 更新操作

```
只更新数据库属性，不操作设备
```

**新规则：** `passport-update` 只更新数据库中的属性字段，不删除和重新添加设备数据

**正确做法：**
```typescript
// 根据 credentialId 查找凭证
const credential = await getCredentialById(credentialId);

// 只更新属性字段
await updateCredentialAttributes(credentialId, {
  show_info: showInfo,
  tags: tags,
  enable: enable,
  auth_model: authModel,
  auth_type_list: authTypeList,
  box_list: boxList,
});
```

#### 2.3 删除操作

```
先删设备成功 → 再删数据库
```

**错误做法：** 同时删除或先删数据库

**正确做法：**
```typescript
// 1. 先删设备
const result = await deleteFromIrisDevice(...);

// 2. 设备成功才删数据库
if (result.success) {
  await deleteCredential(...);
}
```

### 三、MQTT IAMS 消息格式

#### 3.1 消息结构

**标准格式（op + data 嵌套）：**
```json
{
  "time": 1705371902981,
  "requestId": "req-xxx",
  "deviceId": "iris-device-001",
  "op": "passport-add",
  "data": {
    "opId": "1",
    "passportVer": "xxx",
    "personId": "111111111111111111",
    "id": 999999,
    "type": 7,
    "content": "...",
    ...
  }
}
```

#### 3.2 操作类型

| op | 说明 | 是否包含 content | 是否操作设备 |
|------|------|------|------|
| `passport-add` | 新增凭证 | 是 | 是 |
| `passport-update` | 更新属性 | **否** | **否** |
| `passport-del` | 删除凭证 | 否 | 是 |

**⚠️ 注意：** 删除操作是 `passport-del`，不是 `passport-delete`

#### 3.3 字段映射

| IAMS 字段 | 内部字段 | 说明 |
|-----------|----------|------|
| `op` | action | 操作类型 |
| `data.id` | credentialId | 凭证ID |
| `data.type` | credentialType | 凭证类型 |
| `data.personId` | personId | 用户编码 |
| `data.authTypeList` | authTypeList（逗号分隔转数组） | 识别类型列表 |

### 五、ID匹配规则

#### 5.1 添加/更新/删除必须使用相同的 personId

| 操作 | personId | credentialId |
|------|----------|--------------|
| 添加 | 固定值 | 可新生成 |
| 更新 | **必须与添加时相同** | 可新生成 |
| 删除 | **必须与添加时相同** | - |

**注意：** 删除时用 `credentialId`（data.id）查数据库

#### 5.2 错误码处理

| errorCode | 含义 | 处理方式 |
|-----------|------|----------|
| 0 | 成功 | 正常处理 |
| 16 | 人员不存在 | 删除时视为成功（目标状态已达成） |
| 5 | 设备未锁定 | 先锁定设备 |
| 6 | 左眼虹膜为空 | 检查数据 |
| 97 | 设备忙 | 等待后重试 |
| 98 | 设备状态异常 | 重置设备 |

## ⚠️ 血泪教训

### 教训1：数据库操作时机错误

**问题描述：**
- 添加时：先保存数据库，设备超时失败，数据库留下脏数据
- 更新时：先更新数据库，设备操作失败，数据库与设备不一致
- 删除时：同时删除，设备失败但数据库已删除

**解决方案：**
- 所有数据库操作必须在设备操作成功后执行
- 添加：设备成功 → 保存数据库
- 更新：只更新数据库属性，不操作设备
- 删除：设备成功 → 删数据库

**涉及文件：**
- `lib/device-sync.ts` 中的 `handlePassportAdd`、`handlePassportUpdate`、`handlePassportDelete`

---

### 教训2：更新流程变更

**原方案（已废弃）：**
```
删除设备 → 删数据库 → 添加设备 → 更新数据库
```

**新方案：**
```
只更新数据库属性，不操作设备
```

**原因：** IAMS 协议的 `passport-update` 不包含 content 字段，只更新属性信息

---

### 教训3：MQTT消息格式错误

**问题描述：**
- 早期使用扁平结构，与IAMS协议不符
- 字段名不一致：`credentialId` vs `data.id`
- 删除操作名错误：`passport-delete` vs `passport-del`

**解决方案：**
- 改用 `op` + `data` 嵌套结构
- 字段映射：`data.id` → `credentialId`
- 删除操作改为 `passport-del`

---

### 教训4：设备ID不匹配

**问题描述：**
- 测试客户端使用 `iris-001`、`palm-001`
- 数据库配置使用 `iris-device-001`、`palm-device-001`
- 导致"设备不存在"错误

**解决方案：**
- 统一设备ID：`iris-device-001`、`palm-device-001`
- 配置位置：`lib/sync-queue.ts` → `DEFAULT_DEVICES`

## 十四、Kiosk认证流程实现（2026-04-02）

### 14.1 认证流程架构

**重要原则**：前端只调用自己的后端API，不直接调用设备API

```
前端 → 后端代理API → 设备
```

**原因**：
- 安全问题：前端直接调用外部API很危险
- CORS问题：跨域请求不好处理
- Electron环境：需要服务器端日志

### 14.2 后端代理API

| 代理API | 目标设备 | 说明 |
|---------|----------|------|
| `/api/device/palm/query` | 掌纹设备 | 使用Node.js http模块，sendData不编码 |
| `/api/device/iris/records` | 虹膜设备 | 使用fetch，查询识别记录 |
| `/api/auth/verify-password` | 数据库 | 验证密码和胁迫码 |
| `/api/auth/verify-palm` | 数据库 | 验证掌纹userId匹配 |

**代码位置**：
- 掌纹代理：`app/api/device/palm/query/route.ts`
- 虹膜代理：`app/api/device/iris/records/route.ts`
- 密码验证：`app/api/auth/verify-password/route.ts`
- 掌纹验证：`app/api/auth/verify-palm/route.ts`

### 14.3 掌纹认证流程（已测试）

**设备指令**：
- `103` - 开始识别
- `102` - 停止识别

**响应code**：
- `100` - 未识别状态，继续轮询
- `200` - 识别成功，返回des(userId)
- `404` - 识别失败

**认证流程**：
```
1. 进入扫描页 → 发送103开始识别
2. 轮询查询状态（2秒间隔）
3. code=100 → 继续轮询
4. code=200 → 暂停轮询 → 发送102停止 → 验证userId匹配
   - 匹配成功 → 跳转成功页
   - 不匹配 → 发送103重新开始 → 继续轮询
5. code=404 → 继续轮询
6. 超时 → 发送102停止 → 显示超时
```

**停止指令发送时机**：
- 识别成功验证用户前
- 用户不匹配重新开始前
- 超时结束时
- 用户点击重试
- 用户点击返回

### 14.4 虹膜认证流程（已测试）

**关键参数：`lastCreateTime`**

虹膜设备 `/records` API 支持分页查询，使用 `lastCreateTime` 参数来只返回比指定时间更新的记录。

```
首次查询：lastCreateTime = 0
后续查询：lastCreateTime = 上次查询最后一条记录的 createTime
```

**轮询机制**：
- 查询 `/records` 接口获取识别记录
- 3秒轮询间隔，10秒HTTP超时
- 时间窗口：startTime = 当前时间 - 3秒
- **关键**：每次查询后更新 `lastCreateTime`，避免重复返回历史记录

**匹配条件**：
- `staffNum === personId`
- `success === true`
- `type === 1`（虹膜类型）

**⚠️ 重要：时间窗口逻辑**

错误做法（会导致每次返回10条历史记录）：
```typescript
// 每次都用 Date.now() 作为 startTime
startTime: Date.now() - 1000
```

正确做法（只返回新产生的记录）：
```typescript
let lastCreateTime = 0; // 首次查询

// 查询参数
{
  startTime: Date.now() - 3000,
  endTime: Date.now(),
  lastCreateTime: lastCreateTime, // 关键参数
  count: 10,
}

// 查询结果有记录时，更新 lastCreateTime
if (data.body && data.body.length > 0) {
  lastCreateTime = data.body[data.body.length - 1].createTime;
}
```

**代码位置**：
- 前端页面：`app/kiosk/iris/page.tsx`
- 后端代理：`app/api/device/iris/records/route.ts`

**问题 2：revalidateSteps 闭包状态不一致导致步骤完成后卡住**

**现象**：掌纹/虹膜识别成功、页面显示"认证成功"，但不跳转下一步，一直停留在成功页面。

**根因**：
```typescript
// handleBiometricComplete
setCompletedSteps(prev => [...prev, step]);  // 更新 state
const isValid = await revalidateSteps();    // ⚠️ 闭包捕获旧 completedSteps
if (!isValid) return;                       // ⚠️ 直接 return，不跳转
```

`revalidateSteps` 的闭包中 `completedSteps` 是调用前的旧值，与刚更新的 `completedSteps` 不一致 → `isConsistent` 为 false → `return false` → 阻断后续跳转。

**修复**：
- `handleBiometricComplete`：移除 `revalidateSteps()` 调用，直接用当前 `steps` 判断下一步
- `handlePasswordSubmit`（正常密码通过分支）：同样移除 `revalidateSteps()` 调用
- 删除整个 `revalidateSteps` 函数（死代码）
- 所有步骤完成后统一用 `steps.indexOf(step)` 判断是跳转下一步还是跳转成功页

**修改的文件**：
| 文件 | 修改内容 |
|------|----------|
| `app/kiosk/combined/page.tsx` | 密码判断移除 includes(9)；移除 revalidateSteps；修复 handleBiometricComplete、handlePasswordSubmit |

### 18.9 通行记录未上报 + 认证步骤提前通过修复（2026-04-18）

**问题描述**：
1. 组合认证完成后，没有通行记录，IAMS 服务器没有收到上报
2. 密码 → 掌纹 → 虹膜的流程中，虹膜刚显示"认证成功"就跳过了，根本没有实际验证

**根因**：
`goToSuccess` 是 `useCallback`，捕获的 `completedSteps` 是闭包快照。当 `handleBiometricComplete` 或 `handlePasswordSubmit` 调用 `setCompletedSteps(prev => [...prev, step])` 后立即调用 `goToSuccess()` 时，React 的 `setState` 尚未生效，闭包中的 `completedSteps` 是**旧值**。

```
密码通过 → setCompletedSteps(['password']) → setCurrentStep('palm')
掌纹通过 → setCompletedSteps(['password', 'palm']) → setCurrentStep('iris')
虹膜通过 → setCompletedSteps(['password', 'palm', 'iris']) → goToSuccess()
  ↓
goToSuccess 闭包里 completedSteps = ['password', 'palm']  ← 少了一步 iris！
```

**修复方案**：
使用 `useRef` 追踪已完成的步骤，保证 `goToSuccess` 读取到的是最新的、同步的值：
```typescript
const completedStepsRef = useRef<AuthStep[]>([]);

// 每次更新 completedSteps 时同步更新 ref
setCompletedSteps(prev => [...prev, step]);
completedStepsRef.current = [...completedStepsRef.current, step];

// goToSuccess 使用 ref 读取
const authTypes = completedStepsRef.current.map(...)
```

**修改的文件**：
| 文件 | 修改内容 |
|------|----------|
| `app/kiosk/combined/page.tsx` | 添加 completedStepsRef；4处同步更新 ref；goToSuccess 使用 ref 读取 |

### 18.1 虹膜认证时间窗口修复

**问题描述**：
虹膜认证轮询查询时，每次都返回10条历史记录，而不是只返回新产生的识别记录。

使用 `lastCreateTime` 参数实现分页查询机制：

1. 首次查询：`lastCreateTime = 0`
2. 查询返回记录后：更新 `lastCreateTime` 为最后一条记录的 `createTime`
3. 后续查询：传递这个 `lastCreateTime`，设备只返回比该时间更新的记录

**修改的文件**：

| 文件 | 修改内容 |
|------|----------|
| `app/kiosk/iris/page.tsx` | 使用 `lastCreateTime` 变量追踪时间，每次查询后更新 |
| `app/api/device/iris/records/route.ts` | 接收并传递 `lastCreateTime` 参数给虹膜设备 |

**关键代码逻辑**：
```typescript

let lastCreateTime = 0; // 首次查询从0开始

  }
}
```

## 二十四、下发记录优化（2026-04-10）

### 24.1 device_type 字段直接存储

**问题描述**：
下发记录中 `device_type` 全部显示为 `unknown`。

**根本原因**：
MQTT 的 `device_id`（`nuoyadev`，与 IAMS 约定的认证终端ID）与 `device_config` 表中的 `device_id`（`iris-device-001` / `palm-device-001`，我们自己给虹膜/掌纹设备起的名字）**不是同一个东西**，所以 `LEFT JOIN device_config ON sl.device_id = dc.device_id` 永远匹配不上。

**解决方案**：
1. 在 `sync_logs` 表添加 `device_type` 列（`VARCHAR(16)`）
2. 创建日志时直接存入设备类型（`iris` / `palm`），不再依赖 LEFT JOIN
3. 查询时直接从 `sync_logs.device_type` 读取

**修改文件**：
| 文件 | 修改内容 |
|------|----------|
| `lib/database.ts` | `ALTER TABLE sync_logs ADD COLUMN device_type VARCHAR(16)` |
| `lib/sync-queue.ts` | `addSyncLog` 新增 `device_type` 参数；`getSyncLogs` SQL 改为直接读取 |
| `lib/mqtt-client.ts` | 调用 `addSyncLog` 时传入 `device_type` |
| `lib/device-sync.ts` | 所有 `addSyncLog` 调用传入 `device.device_type` |
| `app/api/devices/simulate-iams/route.ts` | 测试模拟 `addSyncLog` 传入 `device_type` |

### 24.2 密码/胁迫码操作不记录到下发记录

**原因**：
密码(type=5)和胁迫码(type=9)是纯数据库操作，不涉及虹膜/掌纹设备下发，不应该出现在设备管理的下发记录中。

**实现**：
```typescript
// 只有虹膜(7)和掌纹(8)才记录到下发记录
const needSyncLog = credentialType === 7 || credentialType === 8;
const queueId = needSyncLog ? await addToSyncQueue({...}) : 0;
```

**下发记录现在只显示**：
- 虹膜添加/删除
- 掌纹添加/删除
- 虹膜设备清空（reset-passport）
- 掌纹设备清空（reset-passport）

### 24.3 reset-passport 清空操作记录到下发记录

**功能**：
收到 IAMS 的 `reset-passport` 指令后，清空虹膜和掌纹设备上的凭证，并将清空操作记录到下发记录。

**实现要点**：
1. 虹膜和掌纹设备各自使用**唯一的 message_id**：`reset-iris-${Date.now()}` 和 `reset-palm-${Date.now()}`
2. 每个设备清空操作包裹在**独立的 try-catch** 中，一个失败不影响另一个
3. 掌纹清空成功后的响应文字简化为 `"已清空"`（107 命令不返回删除数量）

**⚠️ 注意：UNIQUE constraint 教训**
- `sync_queue.message_id` 有 UNIQUE 约束
- 如果虹膜和掌纹清空使用相同的 `message_id`（如 `message.requestId`），第二次插入会报 `SQLITE_CONSTRAINT: UNIQUE constraint failed: sync_queue.message_id`
- **必须保证 message_id 唯一**

**统一格式**：`东八区时间 [类型] 做了什么 成功/失败`

**东八区时间函数**：

### 24.5 重要注意事项

1. **IAMS device_id ≠ 虹膜/掌纹 device_id**
   - IAMS 的 `deviceId`（如 `nuoyadev`）是认证终端ID，用于 MQTT 通信

## 二十五、虹膜设备冷却机制 + 服务器日志查看（2026-04-15）

### 25.1 虹膜设备冷却机制

**背景**：

虹膜设备的 `memberSaveState` 接口（锁定/解锁）偶尔会失败。如果失败后立即返回失败给 IAMS，IAMS 会立即重新下发同一条虹膜凭证，导致设备被连续请求打垮，甚至陷入无限重启。

**核心思路**：给 IAMS 一个"正在处理中"的假象，占用住请求，而不是秒回失败让 IAMS 疯狂重发。

**实现位置**：`lib/device-sync.ts`

#### 全局状态

```typescript
type IrisDeviceState = 'idle' | 'busy' | 'cooling_down';
let irisDeviceState: IrisDeviceState = 'idle';
const IRIS_COOLDOWN_MS = 10000; // 冷却时间 10 秒
```

#### 状态机

```
idle (空闲)
  │
  ├─ 收到请求 → busy → 正常执行：锁定 → 上传 → 解锁
  │              ├─ 成功 → idle
  │              ├─ 锁定失败 → 立即返回失败 → idle
  │              └─ 解锁失败 → sleep 5秒 → 重试解锁
  │                   ├─ 成功 → 返回原操作结果 → idle
  │                   └─ 仍失败 → cooling_down（10秒后自动→idle）
  │
  └─ 冷却期间收到请求 → 接住 → sleep 10秒（不碰设备）→ 返回失败
     （IAMS 10秒后才收到失败响应 → 再重试时设备大概率已恢复）
```

#### 行为说明

| 场景 | 行为 | 对 IAMS 的效果 |
|------|------|----------------|
| 正常操作 | 锁定 → 等待8秒 → 上传 → 等待500ms → 解锁 | 正常返回成功/失败 |
| 锁定失败 | 立即返回失败 | IAMS 收到失败，可重试 |
| 上传失败 | 正常返回失败结果，尝试解锁 | IAMS 收到失败 |
| 解锁失败（首次） | sleep 5秒 → 重试解锁 → 成功则返回原结果 | 占用 IAMS 5秒+，让设备喘息 |
| 解锁失败（重试仍失败） | 进入 10 秒冷却 → 返回失败 | IAMS 收到失败 |
| 冷却期间收到新请求 | 接住请求 → sleep 10秒（不碰设备）→ 返回失败 | 占用 IAMS 10秒，再重试时设备大概率已恢复 |

#### 关键设计原则

1. **冷却不是拒收**：冷却期间仍然接收 IAMS 指令，但 sleep 10 秒后再返回失败
2. **占用 IAMS 等待时间**：让 IAMS 以为"正在处理中"，而不是立即返回失败触发重发
3. **只针对虹膜设备**：掌纹设备不需要（掌纹没有锁定/解锁机制）

## 二十七、设备轮询优化 + ECONNRESET 容错（2026-04-16）

### 27.1 问题背景

## 二十八、虹膜设备上传凭证的完整技术规范（2026-04-16）

### 28.1 虹膜设备 memberSave 接口的完整请求格式

**接口**：`POST http://{设备IP}:{端口}/memberSave`

**请求头**：
```
Content-Type: application/json
```

**请求体（12个固定字段）**：
```json
{
  "staffNum": "用户编码（personId）",
  "cardNum": "",
  "cardType": 0,
  "faceImage": "人脸图片 base64（必须使用 face_photo_sample.txt）",
  "leftIrisImage": "左眼虹膜图片 base64（BMP 8位灰度格式）",
  "rightIrisImage": "右眼虹膜图片 base64（BMP 8位灰度格式）",
  "name": "人员姓名",
  "openDoor": 1,
  "purview": 30,
  "purviewEndTime": 0.0,
  "purviewStartTime": 0.0,
  "singleIrisAllowed": 0
}
```

**关键字段说明**：
- `staffNum`：人员唯一标识，不同人员必须不同
- `faceImage`：必须从 `data/face_photo_sample.txt` 读取，不能为空、不能重复（设备不校验但会提示 Face repetitive）
- `leftIrisImage` / `rightIrisImage`：**必须是 BMP 8位灰度格式的 base64**，不能直接发 JPEG/PNG

### 28.2 人脸照片的获取规则（重要！）

**人脸照片固定使用 `data/face_photo_sample.txt`**：

```typescript
// 唯一正确做法（lib/device-sync.ts）
function getSampleFaceImage(): string {
  const filePath = join(process.env.DATA_DIR || process.cwd(), 'data', 'face_photo_sample.txt');
  try {
    if (existsSync(filePath)) {
      return readFileSync(filePath, 'utf-8').trim();
    }
    return '';
  } catch {
    return '';
  }
}

// MQTT 下发时调用
faceImage: getSampleFaceImage(),

// 测试按钮（app/api/test/iris-upload/route.ts）
const faceFilePath = join(dataDir, 'face_photo_sample.txt');
if (existsSync(faceFilePath)) {
  requestData.faceImage = readFileSync(faceFilePath, 'utf-8').trim();
}

// 测试按钮（app/api/test/iris-add/route.ts）
const faceFilePath = join(process.cwd(), 'data', 'face_photo_sample.txt');
if (existsSync(faceFilePath)) {
  faceImage = readFileSync(faceFilePath, 'utf-8').trim();
}
```

**打包后必须确保**：
1. `face_photo_sample.txt` 文件必须打包到应用安装目录中
2. 打包脚本必须复制 `data/face_photo_sample.txt` 到安装目录
3. 代码中使用 `process.env.DATA_DIR || process.cwd()` + `data/face_photo_sample.txt` 来定位文件
4. Electron 打包后 `DATA_DIR` 指向安装目录，文件应该在安装目录的 `data/` 子目录下

### 28.3 虹膜图片的格式要求（重要！）

**虹膜设备 memberSave 接口只接受 BMP 8位灰度格式的图片。**

**数据源格式差异**：
| 数据来源 | 格式 | 需要转换 |
|----------|------|----------|
| MQTT 下发的 `content` 字段 | JPEG base64 | **是**，需通过 `convertToBmpBase64()` 转换 |
| `members` 接口返回的 iris 数据 | JPEG base64 | **是** |
| 之前成功上传后保存的数据 | BMP base64 | **否**，已经是正确格式 |

**转换函数**（`lib/device-sync.ts` → `convertToBmpBase64()`）：
- 使用 `sharp` 库解码 JPEG/PNG → 灰度 → 构建 BMP 文件
- 8位灰度 BMP：14字节文件头 + 40字节DIB头 + 1024字节调色板 + 像素数据
- 转换结果与设备直接接受的格式完全一致

**关键发现**：
- `iris_test.json` 的 `leftIrisImage` 前10字符是 `Qk02tAQAAA` → 解码为 `BM` = **BMP 格式**（已转换）
- `iris_user_*.json` 的 `irisLeftImage` 前10字符是 `/9j/4AAQSk` → 解码为 `FFD8` = **JPEG 格式**（未转换）
- 如果直接把 JPEG base64 发到 memberSave，设备会拒绝（errorCode 可能为 9 或其他错误）

### 28.4 HTTP 请求方式（重要！）

**必须使用 Node.js 原生 `fetch()` API**：

```typescript
const response = await fetch(`${endpoint}/memberSave`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify(requestData),
  signal: AbortSignal.timeout(15000),
});
const responseData = await response.json();
```

**为什么不用 `http.request()`？**
- `http.request()` 发送大 JSON body（>500KB）时，虹膜设备会主动关闭连接（`other side closed` / `ECONNRESET`）
- 设备端对 `http.request` 和 `fetch` 的处理可能不同，实测 `http.request` 对大数据不稳定
- `fetch` 自动管理 `Content-Length` 和连接，更稳定

**对比结果**：
| 方式 | 小数据（<300KB） | 大数据（>500KB） |
|------|------------------|------------------|
| `fetch()` | ✅ 成功 | ✅ 成功 |
| `http.request()` + `agent: false` | ✅ 成功 | ❌ 设备主动断连 |

### 28.5 完整的虹膜添加流程

```
1. 收到 IAMS MQTT 消息（passport-add，type=7）
       ↓
2. 解析 content 字段，拆分为 leftIrisImage 和 rightIrisImage
   格式："{左眼base64}|==BMP-SEP==|{右眼base64}"
       ↓
3. 通过 convertToBmpBase64() 将 JPEG 转 BMP 8位灰度
       ↓
4. 读取 face_photo_sample.txt 获取人脸照片
       ↓
5. 组装 12 字段请求体（见 28.1）
       ↓
6. 锁定设备（POST /memberSaveState，state=1）
       ↓
7. 等待 8 秒
       ↓
8. 上传人员（POST /memberSave，用 fetch，body=JSON.stringify）
       ↓
9. 等待 500ms
       ↓
10. 解锁设备（POST /memberSaveState，state=0）
       ↓
11. 设备返回 errorCode=0 → 保存数据库
    设备返回 errorCode≠0 → 不保存数据库，返回错误
```

### 28.6 虹膜设备冷却机制（防止打爆设备）

**全局状态**：
```typescript
type IrisDeviceState = 'idle' | 'busy' | 'cooling_down';
let irisDeviceState: IrisDeviceState = 'idle';
const IRIS_COOLDOWN_MS = 10000; // 10 秒
```

**状态机**：
```
idle → busy → 正常流程 → idle
idle → busy → 解锁失败 → retry → 成功 → idle
idle → busy → 解锁失败 → retry → 失败 → cooling_down (10s) → idle
cooling_down → 接住新请求 → sleep 10秒 → 返回失败 → idle
```

**目的**：给 IAMS "正在处理中" 的假象，避免 IAMS 秒收失败后立即重发打爆设备。

## 二十九、数据库查询优化（2026-04-17）

### 29.1 问题：`SELECT *` 导致磁盘 200MB/s 读取

**现象**：现场电脑点击设备管理后，nodejs 进程磁盘读取速度达 200MB/s，系统卡顿。

**根因**：`credentials` 表的以下字段存储 Base64 编码的大数据：

| 字段 | 内容 | 典型大小 |
|------|------|----------|
| `content` | 虹膜 BMP Base64 / 掌纹特征 / 密码 | 1-3MB（虹膜） |
| `iris_left_image` | 左眼虹膜图片 | 1-3MB |
| `iris_right_image` | 右眼虹膜图片 | 1-3MB |
| `palm_feature` | 掌纹特征数据 | 几十KB |

任何 `SELECT * FROM credentials` 都会从磁盘读取这些大字段的完整内容。列表查询时加载全部字段，导致大量磁盘 IO。

### 29.2 解决方案：查询时排除大字段

**核心原则**：数据库的图片/特征字段只用于写入（upsert），读取场景不需要加载：
- 存在性检查：只需要知道有无记录
- 类型查询：只需要 `type` 字段
- 密码验证：只需要 `content`（密码明文）
- 掌纹验证：只需要 `person_id`、`person_name`、`box_list`
- 列表展示：只需要基本信息

**实现**：`lib/db-credentials.ts` 定义轻字段常量：

```typescript
const LIGHT_COLUMNS = 'id, person_id, person_name, person_type, credential_id, type, show_info, tags, auth_model, auth_type_list, box_list, custom_id, enable, created_at, updated_at';
```

| 方法 | 默认查询 | 特殊选项 |
|------|----------|----------|
| `getCredentialById(id)` | 轻字段 | `{ includeContent: true }` |
| `getCredentialsByPersonId(id)` | 轻字段 | `{ includeContent: true }` |
| `getCredentialByPersonId(id)` | 轻字段 | 不需要 |
| `getCredentialByCustomId(id)` | 轻字段 | 不需要 |
| `getAllCredentials()` | 轻字段（默认） | `excludeLarge: false` |

**需要完整字段的场景**：
- `verify-password` API：需要 `content` 字段比对密码，传 `{ includeContent: true }`
- 虹膜新增/更新/删除（device-sync.ts）：数据来自 MQTT payload，不是从数据库查

### 29.3 faceImage 处理

faceImage（人脸照片）不存储在数据库中。虹膜设备下发时从 `getSampleFaceImage()` 读取 `data/face_photo_sample.txt` 添加，内容由任意照片即可。

## 三十、服务断开检测与自动恢复（2026-04-17）

### 30.1 问题

当 Next.js 后台服务意外崩溃或被任务管理器关闭时，Electron 窗口仍能显示页面，但所有 API 调用都会失败。用户无感知，之前的"重启后台服务"按钮也无法正常工作（因为进程引用未清除）。

### 30.2 解决方案

#### 前端：ServiceMonitor 组件

**文件**：`app/ServiceMonitor.tsx`

**工作原理**：
- 每 **30 秒** ping `/api/iams/status`，超时 5 秒
- **连续 2 次失败**才弹窗（避免偶发网络抖动）
- 弹窗显示"后端服务断开，是否启动？"
- 点击"启动服务"调用 `window.electronAPI.restartBackend()`
- 启动成功后 15 秒自动刷新页面
- 非 Electron 环境（浏览器开发模式）自动跳过

**挂载位置**：`app/layout.tsx`，全局生效

#### 后端：修复重启逻辑

**文件**：`electron/main.ts`

**修复点**：
1. `nextProcess.on('exit')` 现在**清除引用**（`nextProcess = null`），使外部杀掉进程后能重新启动
2. `restartEmbeddedServices()` 现在返回实际状态：
   - 检查 Next.js CLI 路径是否存在
   - 等待服务就绪是否成功（40 秒超时）
   - 返回 `{ success: boolean, message: string }`
3. IPC handler `restart-backend` 返回实际结果，不再永远返回 `{ success: true }`

### 30.3 服务生命周期

```
Electron 启动
  → killProcessOnPort(3001)     // 清理旧进程
  → spawn(next start -p 3001)   // 启动新进程
  → waitForServiceReady()       // 轮询 HTTP，最多 20 次×2秒
  → 创建窗口

服务异常退出
  → nextProcess.on('exit') → nextProcess = null
  → ServiceMonitor 检测到（30s 轮询×2 次失败）
  → 弹窗提示
  → 用户点击"启动服务"
  → killProcessOnPort(3001)     // 清理残留
  → spawn(next start -p 3001)
  → waitForServiceReady()
  → 成功 → 刷新页面
```

## 三十一、大字段不持久化优化（2026-04-17）

### 31.1 问题：生产环境数据库膨胀到 10.96GB

**现象**：现场设备的数据库文件达到 10.96GB，但本地测试数据库只有 26MB。

**根因分析**：
- `sync_queue` 表 7,186 行，每行 `payload` 字段存储完整 IAMS 消息（含 `content` 图片数据 ~800KB）
- `credentials` 表虽然只有 2 行，但每行包含 ~1.6MB 的虹膜图片 Base64
- SQLite DELETE 不会缩小数据库文件，只标记页为可复用

**具体占用**：
| 表 | 行数 | 大字段总大小 |
|------|------|----------|
| credentials | 2 | 1.62 MB（iris_left_image + iris_right_image） |
| sync_queue | 7,186 | ~5.7 GB（每条 payload 含完整 content 图片数据） |
| sync_logs | 7,186 | 较小 |

### 31.2 解决方案：不存储大字段

**核心原则**：所有数据转发都是透传，不需要持久化图片/特征数据。

| 凭证类型 | content 用途 | 是否存储 content | 是否存储 iris/palm 字段 |
|----------|-------------|-----------------|----------------------|
| 5 (密码) | 密码明文 | ✅ 存储 | N/A |
| 7 (虹膜) | 虹膜图片 Base64 | ❌ 不存储 | ❌ 不存储 iris_left/right_image |
| 8 (掌纹) | 掌纹特征数据 | ❌ 不存储 | ❌ 不存储 palm_feature |
| 9 (胁迫码) | 胁迫码明文 | ✅ 存储 | N/A |

### 31.3 修改内容

#### 1. `lib/device-sync.ts` — handlePassportAdd 虹膜新增

去掉大字段，只存轻字段：
```typescript
await upsertCredential({
  person_id: payload.personId,
  person_name: memberName,
  credential_id: payload.credentialId,
  type: payload.credentialType as CredentialType,
  auth_type_list: payload.authTypeList?.join(',') || String(payload.credentialType),
  // content, iris_left_image, iris_right_image 不再存储
});
```

#### 2. `lib/device-sync.ts` — handlePassportAdd 掌纹新增

从 `content` 解析 `userId:featureData`，存储 `content` 到数据库，`custom_id` 存 userId：
```typescript
const { userId, featureData } = parsePalmContent(payload.content || '');
syncToPalmDeviceMQTT(device.endpoint, { userId, featureData });
await upsertCredential({
  person_id: payload.personId,
  person_name: payload.personName || payload.personId,
  credential_id: payload.credentialId,
  type: payload.credentialType as CredentialType,
  content: payload.content,  // 完整存储 "userId:featureData"
```

## 三十二、缓存清理功能与 IAMS attr-set 处理优化（2026-04-18）

### 32.1 缓存清理按钮

**问题**：现场设备凭证管理页面"操作"列缺失，怀疑是缓存导致前端加载了旧版本 JS。

**解决方案**：在系统设置页面添加"缓存清理"工具卡片，提供两个按钮：

1. **清理缓存**：通过 IPC 调用 `clear-cache` handler，同时清理：
   - `.next/cache/` 目录（Next.js 编译缓存）
   - `mainWindow.webContents.session.clearCache()`（Electron 浏览器缓存）
2. **刷新页面**：调用 `window.electronAPI.reload()` 刷新当前窗口

**涉及文件**：
- `electron/main.ts`：新增 `clear-cache` IPC handler
- `electron/preload.ts`：暴露 `clearCache` API
- `app/dashboard/settings/page.tsx`：添加缓存清理 UI

**约束**：清理缓存不影响用户数据（settings.json、noah-ark.db、client-config.json 等）。

### 32.2 IAMS attr-set 配置下发处理优化

**问题**：原代码对所有 attr-set 字段统一保存，不返回 `result` 结构，不符合 IAMS 5.3 协议。

**解决方案**：

- 逐个字段处理，每个字段独立保存
- `deviceTime`：尝试用 PowerShell `Set-Date` 设置 Windows 系统时间（5 秒超时，失败忽略），不存数据库
- `passportVer`：持久化到 `device_attrs` 表，重启后可获取
- `model`、`doorModel`、`passRulerList`、`warnRulerList`：存入 `device_attrs` 表
- **永远返回 `code: 200`**，`result` 包含本次请求中设置的所有字段值（原样回显）

**响应格式**：
```json
{
  "time": 1705371902981,
  "requestId": "...",
  "deviceId": "...",
  "op": "attr-set",
  "data": {
    "code": 200,
    "msg": "",
    "result": { "passportVer": "...", "model": 1 }
  }
}
```

**涉及文件**：`lib/mqtt-client.ts`（约第 892 行起的 `attr-set` handler）

### 32.3 版本号 API 修复

**问题**：系统设置中版本号始终显示 0。

**根因**：`/api/version` 读取 `join(process.cwd(), 'electron', 'package.json')`，打包后该路径不存在。

**修复**：使用候选路径遍历策略，先尝试 `electron/package.json`（开发模式），再尝试 `package.json`（打包后模式）。

### 32.4 打包配置修复

**问题**：electron-builder 25.x 开启 asar 打包时 `package.json` 无法被正确包含到 asar 中。

**修复**：设置 `"asar": false`，使用未打包的资源目录模式，安装包大小约 200MB。

#### 3. `lib/device-sync.ts` — handlePassportAdd 掌纹新增 & handlePassportDelete 掌纹删除
