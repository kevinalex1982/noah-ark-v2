## ⚠️ 核心规则（必须遵守）

### 一、虹膜设备操作时序规范

#### 1.1 虹膜添加流程

```
锁定设备 → 等待1秒 → 上传人员(memberSave) → 等待500ms → 解锁设备
```

**时间要求：**
- 锁定后等待：**1秒**
- 上传后等待：**500ms**

**代码位置：** `lib/device-sync.ts` → `syncToIrisDevice()`

#### 1.2 虹膜更新流程

```
只更新数据库属性，不操作设备！
```

**⚠️ 重要变更：** `passport-update` 操作不再操作设备，只更新数据库中的属性字段

**代码位置：** `lib/device-sync.ts` → `handlePassportUpdate()`

#### 1.3 虹膜删除流程

```
直接调用 memberDelete，不需要锁定！
```

**注意：** 删除操作不需要锁定设备

**代码位置：** `lib/device-sync.ts` → `deleteFromIrisDevice()`

---

### 二、数据库操作时机（重要！）

#### 2.1 添加操作

```
先同步设备成功 → 再保存数据库
```

**错误做法：** 先保存数据库，后同步设备（设备失败时数据库有脏数据）

**正确做法：**
```typescript
// 1. 先同步设备
const result = await syncToIrisDevice(...);

// 2. 设备成功才保存数据库
if (result.success) {
  await upsertCredential(...);
}
```

#### 2.2 更新操作

```
只更新数据库属性，不操作设备
```

**新规则：** `passport-update` 只更新数据库中的属性字段，不删除和重新添加设备数据

**正确做法：**
```typescript
// 根据 credentialId 查找凭证
const credential = await getCredentialById(credentialId);

// 只更新属性字段
await updateCredentialAttributes(credentialId, {
  show_info: showInfo,
  tags: tags,
  enable: enable,
  auth_model: authModel,
  auth_type_list: authTypeList,
  box_list: boxList,
});
```

#### 2.3 删除操作

```
先删设备成功 → 再删数据库
```

**错误做法：** 同时删除或先删数据库

**正确做法：**
```typescript
// 1. 先删设备
const result = await deleteFromIrisDevice(...);

// 2. 设备成功才删数据库
if (result.success) {
  await deleteCredential(...);
}
```

### 三、MQTT IAMS 消息格式

#### 3.1 消息结构

**标准格式（op + data 嵌套）：**
```json
{
  "time": 1705371902981,
  "requestId": "req-xxx",
  "deviceId": "iris-device-001",
  "op": "passport-add",
  "data": {
    "opId": "1",
    "passportVer": "xxx",
    "personId": "111111111111111111",
    "id": 999999,
    "type": 7,
    "content": "...",
    ...
  }
}
```

#### 3.2 操作类型

| op | 说明 | 是否包含 content | 是否操作设备 |
|------|------|------|------|
| `passport-add` | 新增凭证 | 是 | 是 |
| `passport-update` | 更新属性 | **否** | **否** |
| `passport-del` | 删除凭证 | 否 | 是 |

**⚠️ 注意：** 删除操作是 `passport-del`，不是 `passport-delete`

#### 3.3 字段映射

| IAMS 字段 | 内部字段 | 说明 |
|-----------|----------|------|
| `op` | action | 操作类型 |
| `data.id` | credentialId | 凭证ID |
| `data.type` | credentialType | 凭证类型 |
| `data.personId` | personId | 用户编码 |
| `data.authTypeList` | authTypeList（逗号分隔转数组） | 识别类型列表 |

### 五、ID匹配规则

#### 5.1 添加/更新/删除必须使用相同的 personId

| 操作 | personId | credentialId |
|------|----------|--------------|
| 添加 | 固定值 | 可新生成 |
| 更新 | **必须与添加时相同** | 可新生成 |
| 删除 | **必须与添加时相同** | - |

**注意：** 删除时用 `credentialId`（data.id）查数据库

#### 5.2 错误码处理

| errorCode | 含义 | 处理方式 |
|-----------|------|----------|
| 0 | 成功 | 正常处理 |
| 16 | 人员不存在 | 删除时视为成功（目标状态已达成） |
| 5 | 设备未锁定 | 先锁定设备 |
| 6 | 左眼虹膜为空 | 检查数据 |
| 97 | 设备忙 | 等待后重试 |
| 98 | 设备状态异常 | 重置设备 |

## ⚠️ 血泪教训

### 教训1：数据库操作时机错误

**问题描述：**
- 添加时：先保存数据库，设备超时失败，数据库留下脏数据
- 更新时：先更新数据库，设备操作失败，数据库与设备不一致
- 删除时：同时删除，设备失败但数据库已删除

**解决方案：**
- 所有数据库操作必须在设备操作成功后执行
- 添加：设备成功 → 保存数据库
- 更新：只更新数据库属性，不操作设备
- 删除：设备成功 → 删数据库

**涉及文件：**
- `lib/device-sync.ts` 中的 `handlePassportAdd`、`handlePassportUpdate`、`handlePassportDelete`

---

### 教训2：更新流程变更

**原方案（已废弃）：**
```
删除设备 → 删数据库 → 添加设备 → 更新数据库
```

**新方案：**
```
只更新数据库属性，不操作设备
```

**原因：** IAMS 协议的 `passport-update` 不包含 content 字段，只更新属性信息

---

### 教训3：MQTT消息格式错误

**问题描述：**
- 早期使用扁平结构，与IAMS协议不符
- 字段名不一致：`credentialId` vs `data.id`
- 删除操作名错误：`passport-delete` vs `passport-del`

**解决方案：**
- 改用 `op` + `data` 嵌套结构
- 字段映射：`data.id` → `credentialId`
- 删除操作改为 `passport-del`

---

### 教训4：设备ID不匹配

**问题描述：**
- 测试客户端使用 `iris-001`、`palm-001`
- 数据库配置使用 `iris-device-001`、`palm-device-001`
- 导致"设备不存在"错误

**解决方案：**
- 统一设备ID：`iris-device-001`、`palm-device-001`
- 配置位置：`lib/sync-queue.ts` → `DEFAULT_DEVICES`

### 14.4 虹膜认证流程（已测试）

**关键参数：`lastCreateTime`**

虹膜设备 `/records` API 支持分页查询，使用 `lastCreateTime` 参数来只返回比指定时间更新的记录。

```
首次查询：lastCreateTime = 0
后续查询：lastCreateTime = 上次查询最后一条记录的 createTime
```

**轮询机制**：
- 查询 `/records` 接口获取识别记录
- 3秒轮询间隔，10秒HTTP超时
- 时间窗口：startTime = 当前时间 - 3秒
- **关键**：每次查询后更新 `lastCreateTime`，避免重复返回历史记录

**匹配条件**：
- `staffNum === personId`
- `success === true`
- `type === 1`（虹膜类型）

**⚠️ 重要：时间窗口逻辑**

错误做法（会导致每次返回10条历史记录）：
```typescript
// 每次都用 Date.now() 作为 startTime
startTime: Date.now() - 1000
```

正确做法（只返回新产生的记录）：
```typescript
let lastCreateTime = 0; // 首次查询

// 查询参数
{
  startTime: Date.now() - 3000,
  endTime: Date.now(),
  lastCreateTime: lastCreateTime, // 关键参数
  count: 10,
}

// 查询结果有记录时，更新 lastCreateTime
if (data.body && data.body.length > 0) {
  lastCreateTime = data.body[data.body.length - 1].createTime;
}
```

### 18.1 虹膜认证时间窗口修复

**问题描述**：
虹膜认证轮询查询时，每次都返回10条历史记录，而不是只返回新产生的识别记录。

**问题原因**：
前端每次查询都使用 `startTime = Date.now() - 1000` 作为时间窗口，虹膜设备会返回该时间范围内的所有历史记录（最多10条）。没有使用 `lastCreateTime` 参数来过滤已查询过的记录。

**解决方案**：
使用 `lastCreateTime` 参数实现分页查询机制：

1. 首次查询：`lastCreateTime = 0`
2. 查询返回记录后：更新 `lastCreateTime` 为最后一条记录的 `createTime`
3. 后续查询：传递这个 `lastCreateTime`，设备只返回比该时间更新的记录

**修改的文件**：

| 文件 | 修改内容 |
|------|----------|
| `app/kiosk/iris/page.tsx` | 使用 `lastCreateTime` 变量追踪时间，每次查询后更新 |
| `app/api/device/iris/records/route.ts` | 接收并传递 `lastCreateTime` 参数给虹膜设备 |

**关键代码逻辑**：
```typescript
// 前端轮询逻辑
let lastCreateTime = 0; // 首次查询从0开始

while (polling) {
  const response = await fetch('/api/device/iris/records', {
    method: 'POST',
    body: JSON.stringify({
      startTime: Date.now() - 3000,
      endTime: Date.now(),
      count: 10,
      lastCreateTime: lastCreateTime, // 关键参数
    }),
  });

  if (data.body && data.body.length > 0) {
    // 更新 lastCreateTime 为最后一条记录的时间
    lastCreateTime = data.body[data.body.length - 1].createTime;
  }
}
```

## 二十、IAMS设备检测功能（2026-04-05）

### 20.1 功能概述

为了让IAMS平台认为我们的设备正常在线，需要实现以下三个功能的联动：

```
┌─────────────────────────────────────────────────────────────────┐
│                        IAMS 平台                                 │
└─────────────────────────────────────────────────────────────────┘
        │                                    │
        │ 5.3 下发配置项                      │ 4 重置凭证库
        ▼                                    ▼
┌─────────────────────────────────────────────────────────────────┐
│                      我们的系统                                   │
│                                                                  │
│  ┌──────────────┐    读取配置     ┌──────────────┐              │
│  │ attr-set     │ ───────────────►│ 状态上报     │──► IAMS      │
│  │ (收到存数据库) │                 │ (每10秒)     │              │
│  └──────────────┘                 └──────────────┘              │
│                                                                  │
│  ┌──────────────┐                                               │
│  │ reset-passport│──► 清空凭证 + 清空版本号                      │
│  └──────────────┘                                               │
└─────────────────────────────────────────────────────────────────┘
```

**重要说明**：这些配置项与我们自己的业务逻辑无关，纯粹是为了通过IAMS的设备在线检测。

### 20.2 功能5.3：设置配置项

**下行主题**：`sys/face/{deviceId}/down/attr-set`

**处理逻辑**：
1. 收到配置项消息
2. 将配置存入数据库 `device_attrs` 表（只有一行）
3. 不需要响应

**配置项内容**（存起来即可，不关心含义）：
| 字段 | 类型 | 说明 |
|------|------|------|
| passportVer | String | 凭证库版本号 |
| model | int | 识别模式 |
| doorModel | int | 门禁模式 |
| passRulerList | Array | 通行规则列表（完整对象） |
| warnRulerList | Array | 告警规则列表（完整对象） |

### 20.3 功能2：设备状态上报

**上行主题**：`sys/face/{deviceId}/up/report-status`

**处理逻辑**：
1. 每10秒执行一次
2. 从数据库 `device_attrs` 表读取配置
3. 组装上报消息发送给IAMS

**消息格式**：
```json
{
  "time": 1705371902981,
  "requestId": "1705371902981",
  "deviceId": "td34jc93",
  "op": "report-status",
  "data": {
    "deviceId": "td34jc93",
    "deviceTime": 1705371902981,
    "ip": "192.168.1.1",
    "passportVer": "td34jc9317053719",
    "model": 1,
    "doorModel": 1,
    "passRulerList": [1, 2, 3, 4],
    "warnRulerList": [1, 2, 3, 4]
  }
}
```

**注意**：`passRulerList` 和 `warnRulerList` 上报时只需要ID列表，不是完整对象。

### 20.4 功能4：重置凭证库

**下行主题**：`sys/face/{deviceId}/down/reset-passport`

**上行主题**：`sys/face/{deviceId}/up/reset-passport`

**处理逻辑**：
1. 收到重置指令
2. 清空 `credentials` 表所有凭证
3. 清空 `device_attrs` 表中的 `passportVer`（设为空字符串）
4. 响应成功消息

**注意**：需要处理重复接收命令的情况
- 已清空时收到命令 → 响应成功
- 正在清空时收到命令 → 忽略，不重复执行

**响应格式**：
```json
{
  "time": 1705371902981,
  "requestId": "1705371902981",
  "deviceId": "td34jc93",
  "op": "reset-passport",
  "data": {
    "code": 200,
    "msg": "",
    "result": {
      "passportVer": ""
    }
  }
}
```

### 20.5 数据库设计

新建 `device_attrs` 表：
```sql
CREATE TABLE device_attrs (
  id INTEGER PRIMARY KEY CHECK (id = 1),  -- 只允许一行
  passportVer TEXT DEFAULT '',
  model INTEGER DEFAULT 1,
  doorModel INTEGER DEFAULT 1,
  passRulerList TEXT DEFAULT '[]',    -- JSON数组
  warnRulerList TEXT DEFAULT '[]',    -- JSON数组
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
)
```

### 20.6 凭证类型处理逻辑（重要）

**核心原则**：
- `deviceId` 是认证终端ID（如 `nuoyadev`），与虹膜/掌纹设备无关
- 只有 `type` 才决定需要操作哪个设备

| type | 类型 | 设备操作 | 数据库操作 |
|------|------|----------|------------|
| 5 | 密码 | 无 | 直接存数据库 |
| 7 | 虹膜 | 虹膜设备 | 设备成功后存数据库 |
| 8 | 掌纹 | 掌纹设备 | 设备成功后存数据库 |
| 9 | 胁迫码 | 无 | 直接存数据库 |

**处理流程**：
```
收到 MQTT 消息
    │
    ├─ type=5 或 9（密码/胁迫码）
    │     └─ 直接操作数据库，不需要物理设备
    │
    ├─ type=7（虹膜）
    │     └─ 查找虹膜设备 → 操作设备 → 成功后存数据库
    │
    └─ type=8（掌纹）
          └─ 查找掌纹设备 → 操作设备 → 成功后存数据库
```

**为什么要这样设计**：
- `authTypeList` 是 IAMS 下发的初始配置，可能与实际凭证不符

| 实际凭证 | type=7(虹膜), type=8(掌纹), type=9(胁迫码) |
| 无密码凭证 | type=5 不存在 |
| **有效认证类型** | 7,8（排除密码5和胁迫码9） |

**显示结果**：
- 单独认证：虹膜认证、掌纹认证（密码不显示）
- 组合认证：显示（虹膜+掌纹=2个有效类型）

## 二十二、胁迫码处理逻辑（2026-04-08）

### 22.1 功能说明

胁迫码是一种特殊凭证，当用户被迫输入胁迫码时：
1. 表面显示认证成功（不让胁迫者察觉）
2. 后台发送告警到 IAMS 平台

### 22.2 触发条件

| 条件 | 是否触发 |
|------|----------|
| 有密码凭证(5) + 有胁迫码(9) + 输入匹配胁迫码 | ✅ 触发告警 |
| 有密码凭证(5) + 无胁迫码(9) | ❌ 正常密码验证 |
| 无密码凭证(5) + 有胁迫码(9) | ❌ 无输入入口，不触发 |

**说明**：如果没有密码凭证，胁迫码没有输入条件，无法触发。

### 22.3 告警消息格式

**主题**：`sys/face/{deviceId}/up/warn-event`

**消息体**：
```json
{
    "time": 1705371902981,
    "requestId": "1705371902981",
    "deviceId": "nuoyadev",
    "op": "warn-event",
    "data": {
        "warnType": 1,
        "passportId": 598,
        "createTime": 1705371902981,
        "warnLevel": 1,
        "warnEventId": 1,
        "warnContent": "胁迫码报警"
    }
}
```

### 22.4 处理流程

```
用户输入密码
    │
```

## 二十四、下发记录优化（2026-04-10）

### 24.1 device_type 字段直接存储

**问题描述**：
下发记录中 `device_type` 全部显示为 `unknown`。

**根本原因**：
MQTT 的 `device_id`（`nuoyadev`，与 IAMS 约定的认证终端ID）与 `device_config` 表中的 `device_id`（`iris-device-001` / `palm-device-001`，我们自己给虹膜/掌纹设备起的名字）**不是同一个东西**，所以 `LEFT JOIN device_config ON sl.device_id = dc.device_id` 永远匹配不上。

**解决方案**：
1. 在 `sync_logs` 表添加 `device_type` 列（`VARCHAR(16)`）
2. 创建日志时直接存入设备类型（`iris` / `palm`），不再依赖 LEFT JOIN
3. 查询时直接从 `sync_logs.device_type` 读取

**修改文件**：
| 文件 | 修改内容 |
|------|----------|
| `lib/database.ts` | `ALTER TABLE sync_logs ADD COLUMN device_type VARCHAR(16)` |
| `lib/sync-queue.ts` | `addSyncLog` 新增 `device_type` 参数；`getSyncLogs` SQL 改为直接读取 |
| `lib/mqtt-client.ts` | 调用 `addSyncLog` 时传入 `device_type` |
| `lib/device-sync.ts` | 所有 `addSyncLog` 调用传入 `device.device_type` |
| `app/api/devices/simulate-iams/route.ts` | 测试模拟 `addSyncLog` 传入 `device_type` |

### 24.2 密码/胁迫码操作不记录到下发记录

**原因**：
密码(type=5)和胁迫码(type=9)是纯数据库操作，不涉及虹膜/掌纹设备下发，不应该出现在设备管理的下发记录中。

**实现**：
```typescript
// 只有虹膜(7)和掌纹(8)才记录到下发记录
const needSyncLog = credentialType === 7 || credentialType === 8;
const queueId = needSyncLog ? await addToSyncQueue({...}) : 0;
```

**下发记录现在只显示**：
- 虹膜添加/删除
- 掌纹添加/删除
- 虹膜设备清空（reset-passport）
- 掌纹设备清空（reset-passport）

### 24.3 reset-passport 清空操作记录到下发记录

**功能**：
收到 IAMS 的 `reset-passport` 指令后，清空虹膜和掌纹设备上的凭证，并将清空操作记录到下发记录。

**实现要点**：
1. 虹膜和掌纹设备各自使用**唯一的 message_id**：`reset-iris-${Date.now()}` 和 `reset-palm-${Date.now()}`
2. 每个设备清空操作包裹在**独立的 try-catch** 中，一个失败不影响另一个
3. 掌纹清空成功后的响应文字简化为 `"已清空"`（107 命令不返回删除数量）

**⚠️ 注意：UNIQUE constraint 教训**
- `sync_queue.message_id` 有 UNIQUE 约束
- 如果虹膜和掌纹清空使用相同的 `message_id`（如 `message.requestId`），第二次插入会报 `SQLITE_CONSTRAINT: UNIQUE constraint failed: sync_queue.message_id`
- **必须保证 message_id 唯一**

### 24.4 日志格式简化

**统一格式**：`东八区时间 [类型] 做了什么 成功/失败`

**东八区时间函数**：
```typescript
function bjt(): string {
  return new Date().toLocaleString('zh-CN', { timeZone: 'Asia/Shanghai', hour12: false });

```

**示例**：
```

2026/4/10 15:03:06 [DeviceSync] ❌ 删除虹膜凭证失败: timeout
```

**修改范围**：
- `lib/mqtt-client.ts` — 所有 console.log 简化
- `lib/device-sync.ts` — 所有 console.log 简化
- `lib/db-device-attrs.ts` — 配置更新日志简化

### 24.5 重要注意事项

1. **IAMS device_id ≠ 虹膜/掌纹 device_id**
   - IAMS 的 `deviceId`（如 `nuoyadev`）是认证终端ID，用于 MQTT 通信
   - 虹膜/掌纹的 `device_id`（如 `iris-device-001`）是我们给下级设备起的名字
   - 两者不能混用，LEFT JOIN 无法匹配

2. **message_id 必须唯一**
   - `sync_queue.message_id` 有 UNIQUE 约束

3. **独立 try-catch**
   - 重置凭证时清空虹膜和掌纹是两个独立操作
   - 每个操作包裹在独立 try-catch 中，互不影响

4. **下发记录只记录设备操作**
   - 密码(5)和胁迫码(9)不下发到设备，不记录
   - 只有虹膜(7)和掌纹(8)的操作才记录

## 二十五、虹膜设备冷却机制 + 服务器日志查看（2026-04-15）

### 25.1 虹膜设备冷却机制

**背景**：

虹膜设备的 `memberSaveState` 接口（锁定/解锁）偶尔会失败。如果失败后立即返回失败给 IAMS，IAMS 会立即重新下发同一条虹膜凭证，导致设备被连续请求打垮，甚至陷入无限重启。

**核心思路**：给 IAMS 一个"正在处理中"的假象，占用住请求，而不是秒回失败让 IAMS 疯狂重发。

**实现位置**：`lib/device-sync.ts`

#### 全局状态

```typescript
type IrisDeviceState = 'idle' | 'busy' | 'cooling_down';
let irisDeviceState: IrisDeviceState = 'idle';
const IRIS_COOLDOWN_MS = 10000; // 冷却时间 10 秒
```

#### 状态机

```
idle (空闲)
  │
  ├─ 收到请求 → busy → 正常执行：锁定 → 上传 → 解锁
  │              ├─ 成功 → idle
  │              ├─ 锁定失败 → 立即返回失败 → idle
  │              └─ 解锁失败 → sleep 5秒 → 重试解锁
  │                   ├─ 成功 → 返回原操作结果 → idle
  │                   └─ 仍失败 → cooling_down（10秒后自动→idle）
  │
  └─ 冷却期间收到请求 → 接住 → sleep 10秒（不碰设备）→ 返回失败
     （IAMS 10秒后才收到失败响应 → 再重试时设备大概率已恢复）
```

#### 行为说明

| 场景 | 行为 | 对 IAMS 的效果 |
|------|------|----------------|
| 正常操作 | 锁定 → 等待1秒 → 上传 → 等待500ms → 解锁 | 正常返回成功/失败 |
| 锁定失败 | 立即返回失败 | IAMS 收到失败，可重试 |
| 上传失败 | 正常返回失败结果，尝试解锁 | IAMS 收到失败 |
| 解锁失败（首次） | sleep 5秒 → 重试解锁 → 成功则返回原结果 | 占用 IAMS 5秒+，让设备喘息 |
| 解锁失败（重试仍失败） | 进入 10 秒冷却 → 返回失败 | IAMS 收到失败 |
| 冷却期间收到新请求 | 接住请求 → sleep 10秒（不碰设备）→ 返回失败 | 占用 IAMS 10秒，再重试时设备大概率已恢复 |

#### 关键设计原则

1. **冷却不是拒收**：冷却期间仍然接收 IAMS 指令，但 sleep 10 秒后再返回失败
2. **占用 IAMS 等待时间**：让 IAMS 以为"正在处理中"，而不是立即返回失败触发重发
3. **只针对虹膜设备**：掌纹设备不需要（掌纹没有锁定/解锁机制）

#### 修改的文件

| 文件 | 修改内容 |
|------|----------|
| `lib/device-sync.ts` | 新增 `irisDeviceState` 全局状态 + `IRIS_COOLDOWN_MS` 常量；`syncToIrisDevice()` 函数增加状态检查、解锁重试、冷却逻辑 |

## 二十七、设备轮询优化 + ECONNRESET 容错（2026-04-16）

### 27.1 问题背景

掌纹设备在连续请求时会返回 `ECONNRESET`（连接重置），说明设备端忙碌时主动断开 TCP 连接。之前存在多个来源同时向掌纹设备发请求的问题：

- `/api/devices` 前端轮询
- `sync-scheduler` 后端定时检查
- `processSyncQueue` 每个队列项预检查

导致掌纹设备被打爆。

### 27.2 设备轮询拆分

**前端改为分开轮询**，通过 `?type` 参数只查询指定设备：

| 项目 | 间隔 | API 调用 | 说明 |
|------|------|----------|------|
| 掌纹设备 | **15 秒** | `GET /api/devices?type=palm` | 记录日志向厂家反映 |
| 虹膜设备 | **30 秒** | `GET /api/devices?type=iris` | 正常检查 |
| IAMS 状态 | **10 秒** | `GET /api/iams/status` | 读数据库缓存 |
| 下发记录 | **10 秒** | `GET /api/devices/sync?action=logs` | 读数据库 |

### 27.3 ECONNRESET 容错

掌纹设备在忙碌时会主动断开 TCP 连接（`read ECONNRESET`），这不是请求格式错误，而是设备端自身的保护机制。

**处理策略**：
- 设备视为**在线**（不标记为离线）
- **不更新**凭证数量（保留上次查询的值）
- 前端不显示"设备 API 无响应"，只显示干净的在线状态
- 下一轮 15 秒后再查

**涉及代码位置**：
| 文件 | 说明 |
|------|------|
| `app/api/devices/route.ts` | `getPalmCredentialCount()` — ECONNRESET 返回 `{ online: true, count: null }` |
| `lib/device-sync.ts` | `checkDeviceStatus()` — ECONNRESET 返回 `{ online: true }` |
| `app/api/devices/status/route.ts` | `checkPalmDeviceHealth()` — ECONNRESET 返回 `{ online: true }` |
